## Successes

- **Architectural discipline from day one** — GameEvents, GameConstants, GameEnums established in Story 1.1; every subsequent story plugged into those patterns without friction. Consistent conventions (InputAction lifecycle, event bus, programmatic creation) drove clean, fast execution.
- **Code review quality trajectory** — Resource cleanup pattern solidified iteratively: InputAction disposal (1.1), material leak (1.2), Sprite leak (1.6), DontDestroyOnLoad (1.7). By Story 1.8, clean review on first pass.
- **User testing feedback drove meaningful polish** — Square crosshair refined to circular dot (1.6), flashlight wall hotspot noted for future tuning (1.8), FPS verification question led to FPSDisplay tool (1.4/1.5).
- **Programmatic creation as default** — Pivoted from scene YAML to code-based creation for complex Unity objects (Canvas in 1.6, Spot Light in 1.8). Cleaner, more maintainable, avoids YAML fragility.
- **Massive performance headroom** — 680–800 FPS with current scene gives room for Epic 2's heavier systems.
- **MVP discipline** — Scope stayed tight. Placeholder systems (save/load) established foundation without over-engineering.

---

## Challenges

- **Resource cleanup was most recurring review finding** — 4 of 8 stories had resource leak issues caught in review. Pattern solidified by end of epic but was a consistent theme.
- **Manual component attachment in Unity Editor** — PerformanceMonitor (1.5), PlayerHUD (1.6), SaveManager (1.7), PlayerFlashlight (1.8) all required user to attach components after import. Fundamental limitation of editing scene YAML outside Editor (script GUID unavailable pre-import). Manageable friction, not debt.
- **Story file conflicts during review phase** — User's manual test notes and task checkmarks were overwritten when the code review agent edited the same story file using cached content. Critical process issue.
- **Dev agent cannot assess visual feel** — Structure validation is reliable, but gameplay polish (crosshair shape, flashlight hotspot) requires human eyes in the Unity Editor.

---

## Key Insights

1. **Architectural discipline drove clean execution** — Consistent patterns from Story 1.1 onward meant every subsequent story slotted in without friction.
2. **Code review quality improved iteratively** — Resource cleanup pattern solidified across the epic, from multiple findings in early stories to clean passes by the end.
3. **Re-read before edit** — Shared story files need fresh reads before every write to prevent user changes from being overwritten.
4. **Human eyes are essential** — Dev agent validates structure, but visual feel and gameplay polish require user verification in the Unity Editor.
5. **Manual Editor steps are acceptable friction** — Keeps user engaged in the development lifecycle, decreases knowledge gaps.

## Action Items

### Process Improvements

1. **Re-read shared files before every edit**
   - Story files are modified by both user and agent. Agent must re-read current file state immediately before writing, never rely on cached content.
   - Owner: Dev Agent (all workflows)
   - Success criteria: Zero file conflict prompts during Epic 2

2. **Resource cleanup checklist**
   - Any story creating runtime resources (InputActions, Materials, Textures, Sprites, GameObjects) must include explicit OnDestroy() cleanup in the task list.
   - Owner: Story creation workflow
   - Success criteria: Zero resource leak findings in Epic 2 code reviews

### Team Agreements

- Manual testing tasks stay unchecked by the dev agent — always
- Programmatic creation is the default for complex Unity objects
- All tunable values in GameConstants — no exceptions
- Visual feel requires human verification — dev agent validates structure, user validates experience
