## Challenges

- **InspectionSystem entry guards growing** — Each new interaction type (2-5 displaced, 2-6 fixture, 2-7 pickup) added guards to InspectionSystem's inspection entry conditions. Works but accumulating complexity.
- **PlayerHUD trending toward God Object** — Owns InspectionUI, DiscoveryNotificationUI, InventoryUI, crosshair, and prompt text by end of epic. Addressed in Epic 3 preparation with NotebookUI as peer system.
- **Manual testing instructions need more expression** — Story instructions for Unity Editor steps need to guide the user toward real, purposeful scene assets with clear context on why and where — not just component attachment checklists.
- **Uncommitted work risk** — Story 2-7 was implemented but session shut down before commit. Work survived only because it was in the working tree.
- **Resource cleanup not yet at zero** — Story 2-4 had event subscription cleanup caught in review. Down from 50% hit rate (Epic 1) to ~14%, but not the zero target.

---

## Key Insights

1. **Convention discipline compounds** — Epic 1's architectural patterns drove fast, confident delivery across all 7 stories. When patterns are clear, fixes are obvious.
2. **Think like tables** — Cross-referencing for Epic 3 is relational data modeling with unique IDs, not fuzzy pattern matching. Get the schema right in Story 3-1 and downstream stories follow naturally.
3. **NotebookUI as a peer, not a child** — Independent system with its own Canvas and lifecycle prevents PlayerHUD God Object trajectory. Communicates via GameEvents, doesn't inherit from PlayerHUD.
4. **Manual testing = real user flows** — Testing instructions should read like player scenarios with intentional scene assets, not component audits. Same philosophy as Playwright tests validating real user journeys.
5. **Express the intention** — Story deliverables must guide the user toward placing real, purposeful assets in the scene. Not hardcoded test fixtures — actual game content the user keeps and builds on.
6. **The team learns and adjusts quickly** — "Context intention" over rigid compliance. Understanding the *why* behind a rule and applying it with judgment.

## Epic 3 Preparation

### Architectural Decisions (Locked In)

- **NotebookManager as independent system** — peer to PlayerHUD, own Canvas, own lifecycle. Communicates via GameEvents (`NotebookOpened`/`NotebookClosed`). PlayerHUD fades out when notebook opens.
- **Relational data model** — unique entry ID, source object ID, zone ID, category enum, timestamp, content. Cross-referencing in Story 3-4 is ID-based lookups against shared identifiers.
- **DiscoveryLog → NotebookManager migration** — clean transition in Story 3-1. Existing auto-log pipeline feeds into the new system. DiscoveryEntry evolves into the richer notebook entry model.

### Technical Prerequisites

- Tab input handling for notebook toggle — research coexistence with existing InputAction system
- No pre-refactoring required — NotebookUI is additive as a peer system

### Critical Blockers

- None. Epic 2 foundation covers all Epic 3 dependencies.
